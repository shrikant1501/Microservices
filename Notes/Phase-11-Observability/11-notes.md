# Phase 11 — Observability in Microservices
## (Correlation IDs · Distributed Tracing · Micrometer · Actuator · Zipkin)

> **80/20 Focus:** The three questions every interviewer asks — "How do you debug a failed request across 5 services?", "What is distributed tracing?", and "What is a trace vs a span?" Master these and the rest is detail.

---

## 1. The Problem

In a monolith, debugging a failed request is trivial:
```
Stack trace → single JVM → one log file → found it.
```

In microservices:
```
Client → API Gateway → todo-service → user-service → [Kafka] → notification-service
```

A request touches **5 processes**, potentially on **5 different machines**, writing to **5 different log files**.

When something goes wrong, you have:
- Gateway logs: "Forwarded POST /api/todos"
- todo-service logs: "Creating todo for user 42"
- user-service logs: "getUserById(42) called"
- notification-service logs: "Consumed event for user 42"

**Without correlation**: you cannot connect these log lines. You don't know which todo-service log belongs to which gateway request.

**The solution**: attach a unique ID to every request at the entry point (gateway), carry it through every service, log it on every line, include it in every async event.

## 3. Pillar 1: Structured Logging with Correlation ID

### What is a Correlation ID?

A **Correlation ID** is a UUID generated at the entry point of a request and carried through every service and every log line for that request's lifetime.

```
Client sends: POST /api/todos
   ↓
Gateway generates: X-Correlation-Id: abc-123
   ↓ (injects into request header)
todo-service receives: X-Correlation-Id: abc-123
   logs: "[abc-123] Creating todo for user 42"
   ↓ (Feign propagates the header)
user-service receives: X-Correlation-Id: abc-123
   logs: "[abc-123] Fetching user 42"
   ↓ (included in Kafka event payload)
notification-service receives: correlationId=abc-123
   logs: "[abc-123] Sending email to user 42"

To debug: grep all logs for "abc-123" → complete picture.
```

### Correlation ID vs Trace ID

| | Correlation ID | Trace ID (Zipkin/OTEL) |
|---|---|---|
| **What** | Custom UUID in header | Auto-generated by tracing lib |
| **Purpose** | Log correlation | Distributed tracing tree |
| **Propagation** | Manual (filter + Feign interceptor) | Automatic via W3C TraceContext |
| **Visualisation** | Text grep / Kibana query | Zipkin UI / Jaeger UI |
| **Overhead** | Zero | Small (sampling rate configurable) |

**In production, you use both.** Correlation ID for log filtering, Trace ID for visual trace trees.

### MDC (Mapped Diagnostic Context)

SLF4J's `MDC` is a **per-thread key-value store** that logging frameworks (Logback, Log4j2) automatically append to every log line.

```java
MDC.put("correlationId", "abc-123");
log.info("Processing request"); // → [abc-123] Processing request
MDC.clear();                    // MUST clear to prevent thread pool leaks
```

In our implementation:
- Gateway injects `X-Correlation-Id` header into every forwarded request
- Each downstream service has a `CorrelationIdMdcFilter` (a `OncePerRequestFilter`)
  that reads `X-Correlation-Id` header → puts in MDC → removed after request

```
logback pattern: %d %-5level [%X{correlationId}] %logger{36} - %msg%n
output:          2024-01-01 INFO [abc-123] c.m.todo.api.TodoService - Creating todo
```

## 4. Pillar 2: Distributed Tracing

### Core Concepts

```
TRACE = One complete request journey across all services
SPAN  = One unit of work within that journey (one service call, one DB query)

EXAMPLE:
Trace ID: xyz-789
├── Span A: Gateway → todo-service           (10ms)
│   ├── Span B: todo-service → user-service  (5ms)
│   └── Span C: todo-service DB insert       (2ms)
└── Span D: Kafka event consumption          (3ms, async child)

Zipkin UI shows this as a Gantt chart → instantly see which span is slowest.
```

### Key Terms

| Term | Definition |
|---|---|
| **Trace** | The complete end-to-end path of a request through all services |
| **Span** | A single named, timed operation (HTTP call, DB query, Kafka produce) |
| **Trace ID** | One UUID shared by all spans in the same trace |
| **Span ID** | Unique ID for each span |
| **Parent Span ID** | ID of the span that created this span (forms the tree) |
| **Baggage** | Key-value pairs propagated with the trace (like Correlation ID) |
| **Sampling Rate** | % of requests that generate traces (e.g. 10% in production) |

### W3C TraceContext Propagation

Modern standard (used by OpenTelemetry):

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │   │                                │                 │
             │   Trace ID (128-bit)               Span ID (64-bit)  Flags
             Version
```

Older Zipkin B3 format (still common):
```
X-B3-TraceId: abc123
X-B3-SpanId: def456
X-B3-ParentSpanId: parent789
X-B3-Sampled: 1
```

### Micrometer Tracing (Spring Boot 3.x)

Spring Boot 3.x unified tracing under **Micrometer Tracing** (replaces Spring Cloud Sleuth from Boot 2.x).

```xml
<!-- Micrometer Tracing bridge to Brave (Zipkin's tracing library) -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<!-- Sends trace data to Zipkin -->
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>
```

Configuration:
```properties
# Sample 100% in dev, 10% in production
management.tracing.sampling.probability=1.0
# Zipkin endpoint
management.zipkin.tracing.endpoint=http://localhost:9411/api/v2/spans
# Auto-injects traceId into MDC — logs show traceId automatically
logging.pattern.correlation=[${spring.application.name:},%X{traceId:-},%X{spanId:-}]
```

**What happens automatically with Micrometer Tracing:**
- Every HTTP request → new Trace + root Span (or continues existing if headers present)
- Every Feign call → new child Span, propagates trace headers to downstream
- Every Kafka produce → injects trace headers into Kafka record headers
- Every `@Async` method → span propagated to the new thread
- Every DB query (with JDBC instrumentation) → child span

## 5. Pillar 3: Metrics with Micrometer + Actuator

### Spring Boot Actuator (What interviewers ask about)

Actuator exposes operational endpoints over HTTP (and JMX).

Key endpoints:

| Endpoint | What it shows |
|---|---|
| `/actuator/health` | Service health (DB, Kafka, disk, custom indicators) |
| `/actuator/info` | App version, git commit, description |
| `/actuator/metrics` | List of all available metric names |
| `/actuator/metrics/{name}` | Specific metric value with tags |
| `/actuator/circuitbreakers` | Resilience4j CB state (CLOSED/OPEN/HALF_OPEN) |
| `/actuator/loggers` | View/change log levels at runtime (NO restart!) |
| `/actuator/env` | All resolved properties (config server, env vars, etc.) |
| `/actuator/refresh` | Trigger @RefreshScope bean reload (Phase 7) |
| `/actuator/prometheus` | Metrics in Prometheus format (needs micrometer-registry-prometheus) |

### Micrometer — the Metrics Facade

Micrometer is to metrics what SLF4J is to logging — a **vendor-neutral facade**.

```
Your code (Micrometer API) → Counter / Gauge / Timer / DistributionSummary
                           ↓
         Prometheus / Datadog / CloudWatch / InfluxDB / Graphite
                           (your choice of backend)
```

**Built-in auto-instrumentation (zero code):**
- JVM memory, GC, threads
- HTTP request count, duration, status codes
- HikariCP connection pool metrics
- Kafka consumer lag
- Resilience4j circuit breaker states

**Custom metric example:**
```java
// Counter — how many todos were created today?
@Autowired MeterRegistry meterRegistry;

Counter todoCreated = Counter.builder("todos.created")
    .tag("service", "todo-service")
    .register(meterRegistry);

// In createTodo():
todoCreated.increment();
```

**Query in Prometheus:**
```promql
rate(todos_created_total[5m])   # todos created per second in last 5 min
```

## 7. Implementation in Our Project

### What we implement (code):
1. `CorrelationIdMdcFilter` in each service — reads `X-Correlation-Id` header → puts in MDC
2. `FeignCorrelationInterceptor` in todo-service — propagates `X-Correlation-Id` on Feign calls
3. Update Logback pattern to include `%X{correlationId}`
4. Add Micrometer Tracing deps + Zipkin reporter (concept wire-up — Zipkin server via Docker in Phase 12)

### What the gateway already does (Phase 6):
- Generates `X-Correlation-Id` if absent
- Passes through existing `X-Correlation-Id`
- Adds it to both request (downstream) and response (client)

### What we DON'T implement (concepts only):
- Full ELK stack (too heavyweight for this learning environment)
- Prometheus + Grafana (Docker Compose in Phase 12 will cover this)
- OpenTelemetry Collector (enterprise concern)

## 8. Common Mistakes

| Mistake | Consequence | Fix |
|---|---|---|
| Not clearing MDC after request | Next request on same thread sees previous request's correlationId | Use `try-finally { MDC.clear() }` |
| Not propagating to async threads | `@Async` methods lose the correlation ID | Use `MDC.getCopyOfContextMap()` and restore in new thread |
| Not including in Kafka events | Async consumers lose the trace | Include `correlationId` in event payload, not just headers |
| Logging raw user data in traces | PII leak in Zipkin/logs | Log only IDs, never email/phone/name in trace spans |
| 100% sampling in production | Overwhelms Zipkin, performance hit | Set `sampling.probability=0.1` (10%) in prod |
| Exposing all actuator endpoints | `/actuator/env` leaks secrets | In prod: only expose `health` and `info` externally |

## 9. Production Best Practices

```
1. STRUCTURED LOGGING
   → Log in JSON format (not plain text) in production
   → Logstash/Fluentd can then parse fields automatically
   → Include: correlationId, traceId, spanId, userId, service, level
   
2. LOG LEVELS BY ENVIRONMENT
   → dev: DEBUG (your packages), INFO (Spring)
   → prod: INFO (your packages), WARN (Spring)
   → Change at runtime: POST /actuator/loggers/com.myapp { "configuredLevel": "DEBUG" }
   
3. SAMPLING STRATEGY
   → Default: 10% sampling (management.tracing.sampling.probability=0.1)
   → Error traces: always sample (100%)
   → Slow traces: always sample (threshold-based sampling)
   
4. HEALTH CHECK DESIGN
   → /actuator/health/liveness  → Is the app running? (K8s liveness probe)
   → /actuator/health/readiness → Is the app ready to serve? (K8s readiness probe)
   → Include DB, Kafka, Redis in readiness (but NOT in liveness)
   
5. CORRELATION ID STANDARDS
   → Use X-Request-ID (industry standard) or X-Correlation-Id (team convention)
   → Generate at the outermost entry point only (gateway)
   → Never expose internal span IDs to clients
```

## 10. Interview Questions — Standard

**Q1: How do you debug a slow request that spans 5 microservices?**

> You need distributed tracing. Each service propagates the trace ID (via `traceparent` header or Zipkin B3). All spans are reported to Zipkin/Jaeger. The Zipkin UI shows you a Gantt chart: which service took the most time, and which DB call or downstream call is the bottleneck. For logs, every service also logs the Correlation ID, so you can `grep correlationId=abc-123` across all log files (or query Kibana/Grafana Loki) to see the complete log trail.

**Q2: What is the difference between a Trace and a Span?**

> A **Trace** is the complete end-to-end journey of a single request through all services — it has one Trace ID shared by everyone. A **Span** is one unit of work within that trace — like "todo-service processed this request" or "DB query ran". Each span has its own Span ID and a Parent Span ID pointing to who triggered it. Spans form a tree under one trace.

**Q3: What is Micrometer?**

> Micrometer is the metrics facade for Spring Boot. Like SLF4J abstracts logging backends, Micrometer abstracts metrics backends (Prometheus, Datadog, CloudWatch). You write `counter.increment()` once, and configure the backend in properties. Spring Boot auto-instruments JVM, HTTP, DB pool, Kafka, Resilience4j — zero code needed for standard metrics.

**Q4: How does correlation ID differ from trace ID?**

> A correlation ID is a custom header we generate and carry — it's great for log correlation but gives no structural insight into timing or call trees. A trace ID is part of distributed tracing (Zipkin/Jaeger/OTEL) and is automatically propagated by the instrumentation library. It enables visual trace trees showing parent-child span relationships. In practice, with Micrometer Tracing, the trace ID IS effectively your correlation ID — the `traceId` is available in MDC automatically.

**Q5: What actuator endpoints do you expose in production?**

> Externally (via API Gateway or load balancer): only `/actuator/health` for health checks. Internally (within the cluster, secured by network policies): `health`, `metrics`, `prometheus`, `info`. Never expose `env`, `configprops`, or `heapdump` externally — they leak configuration and credentials.

## 11. Tricky Interview Questions

**Q: SLF4J MDC is ThreadLocal-based. What breaks in a reactive (WebFlux) application?**

> In reactive code (like our API Gateway which is WebFlux-based), a single request can execute across multiple threads. `ThreadLocal`/MDC values are lost when the reactive pipeline switches threads. The solution is **Reactor's `Context`** (via `contextWrite` and `Mono.deferContextual()`), which propagates with the reactive chain regardless of thread switches. Alternatively, Micrometer's `Observation` API and Reactor's `ContextSnapshot` handle this automatically.

**Q: If you have a 10% sampling rate and a customer complains about a specific failed request, how do you find the trace?**

> If that specific request was in the 90% not sampled, it won't be in Zipkin. This is why you combine two approaches: (1) **error-based sampling** — always sample requests that result in errors (5xx, circuit breaker trips), regardless of the global rate. (2) **Correlation ID logging** — even unsampled requests are logged with the correlation ID. The customer provides the correlation ID from the error response header, and you search the centralized log system (ELK/Loki) with it.

**Q: How does tracing work across Kafka (asynchronous boundaries)?**

> The producer injects trace headers into the Kafka record's **headers** (not the payload). The consumer library (with tracing instrumentation) reads these headers, extracts the trace context, and creates a new span as a child of the producer's span. The async gap is represented in Zipkin as a link between the producer span and the consumer span. Without this, async operations appear as separate, unrelated traces.

## 12. Scenario-Based Questions

**Scenario 1:** "Our p99 latency spiked from 200ms to 2s during peak hours. How do you investigate?"

> 1. Prometheus/Grafana → look at `http_server_requests_seconds` histogram by service → find which service's p99 spiked
> 2. Zipkin → filter traces by duration > 1s during that time window → find which span is slowest
> 3. Look for: DB slow queries (check HikariCP pool exhaustion), Kafka consumer lag, downstream service timeouts, GC pauses (JVM metrics in Grafana)
> 4. Circuit breaker state → was any CB in HALF_OPEN during that window?
> 5. Correlation ID → pull sample slow request logs → see the exact path

**Scenario 2:** "A downstream service doesn't support W3C TraceContext headers, only B3 format. What do you do?"

> Micrometer Tracing supports both. Configure `management.tracing.propagation.type=B3` or `W3C` (or both for backward compatibility: `B3,W3C`). The tracing library automatically reads/writes both formats. This is a common scenario during migration from Spring Cloud Sleuth (B3) to OpenTelemetry (W3C).

## 13. Quick Revision Cheat Sheet

```
CORRELATION ID
  → UUID generated at Gateway
  → Carried in X-Correlation-Id header
  → Each service: reads header → MDC.put() → logs it → MDC.clear()
  → Feign interceptor propagates to downstream calls
  → Kafka events carry it in payload

DISTRIBUTED TRACING
  → Trace = complete request journey (one Trace ID)
  → Span = one unit of work (own Span ID + Parent Span ID)
  → Micrometer Tracing (Spring Boot 3) = replaces Spring Cloud Sleuth
  → Zipkin = visualises trace trees, shows timing Gantt chart
  → W3C traceparent header = modern standard
  → Sampling: 10% prod, 100% dev, 100% errors always

MICROMETER
  → Metrics facade (like SLF4J for logs)
  → Auto-instruments: HTTP, JVM, DB pool, Kafka, Resilience4j
  → Backends: Prometheus, Datadog, CloudWatch, etc.
  → counter.increment() / timer.record() / gauge.value()

ACTUATOR
  → /health → liveness + readiness (K8s probes)
  → /metrics → all metric names
  → /loggers → change log level at runtime (no restart)
  → /circuitbreakers → CB state
  → /prometheus → Prometheus scrape endpoint
  → PRODUCTION: only expose health externally

MDC vs Reactor Context
  → MDC: works for servlet (thread-per-request)
  → Reactor Context: required for WebFlux (reactive, multi-thread)
  → Micrometer Observation API: works in both

TOOLING STACK (Production)
  → Logs: JSON → Fluentd/Logstash → Elasticsearch → Kibana
  → Metrics: Micrometer → Prometheus → Grafana
  → Traces: Micrometer Tracing → Zipkin/Jaeger
  → All-in-one: OpenTelemetry Collector (modern approach)
```
